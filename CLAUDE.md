# CLAUDE.md

## שפה וכיווניות — עברית בלבד, RTL (מחייב)

האתר הוא **אתר עברי, בעברית בלבד, ובכיוון ימין-לשמאל (RTL)**. אין גרסה אנגלית/רב-לשונית.

חוקיות:
- כל עמוד: `<html lang="he" dir="rtl">`. כל התוכן, ה‑UI, ה‑placeholders, ה‑alt וה‑aria — בעברית.
- העיצוב, הפריסות והרכיבים מתוכננים RTL-first (יישור, סדר עמודות/טאבים, אייקוני כיוון, מרווחים).
- מונחים לועזיים (Influ Capital, ETF, SEO וכו') מותרים בתוך טקסט עברי; אסור לבנות ממשק/ניווט באנגלית.
- אין להוסיף תשתית i18n/בורר שפה אלא אם המשתמש ביקש במפורש.

## עקרונות יסוד לבנייה (לא ניתנים למשא ומתן כברירת מחדל)

כל בנייה בפרויקט, לאורך כל התהליך, חייבת להיות מותאמת באופן מושלם ל:

- **SEO** — מבנה סמנטי, תגיות meta, נתונים מובנים (Schema), כותרות תקינות, קישוריות פנימית.
- **GEO** (Generative Engine Optimization) — תוכן ברור ומובנה שמנועי AI יכולים לצטט ולהבין.
- **פרטיות** — מינימום איסוף נתונים, ללא טראקרים מיותרים, התאמה לתקנות (GDPR / חוק הגנת הפרטיות).
- **נגישות** (a11y) — תאימות WCAG, ניגודיות, ניווט מקלדת, תגיות ARIA, תמיכה בקוראי מסך.
- **מהירות וביצועים** — קוד רזה, אופטימיזציית נכסים, Core Web Vitals טובים, טעינה מהירה.

חוקיות:
- **תמיד** לקבל החלטות שנאמנות לעקרונות אלה כברירת מחדל, גם כשלא התבקש במפורש.
- אם המשתמש מבקש דרישה שפוגעת באחד מהעקרונות — **לעצור, להסביר בקצרה למה לא כדאי**, ולתת לו להחליט אם להמשיך בכל זאת (ההחלטה הסופית שלו).

## ריכוזיות ומקור אמת יחיד (Single Source of Truth) — עיקרון על

הפרויקט נבנה **חכם, עם assets ו‑tokens מרוכזים**. אסור לפזר ערכים קשיחים (hard-coded) בקוד. כל ערך — צבע, מידה, פינה, הצללה, פונט, לוגו, קופי, פרט קשר — נשאב ממקור אמת יחיד. שינוי במקור מתעדכן אוטומטית בכל מקום. שינויים נקודתיים — רק אם המשתמש מבקש במפורש.

מקורות האמת של הפרויקט:

| תחום | מקור האמת | איך משתמשים |
|------|-----------|-------------|
| **Design Tokens + רכיבי ליבה** (צבעים, מרווחים, פינות, הצללות, מוטשן, כפתורים, טפסים, תגיות) | **`brand.css`** (`:root` + מחלקות) | לטעון `brand.css` ולהשתמש ב‑`var(--token)` ובמחלקות הקיימות בלבד. אסור hex/px קשיח ואסור לשכפל CSS. סגנונות תצוגה‑בלבד (doc‑chrome של עמודי הראווה — סיידבר, ויזואליזציות צבע/טיפוגרפיה/לוגו) יושבים ב‑`brand-docs.css` ונטענים רק ע״י brand-book/heroes/sections/design-verticals; `index.html` טוען רק `brand.css` (ביצועים). |
| **תיעוד שפת המותג** (תצוגת הצבעים, טיפוגרפיה, לוגו, שפה וטון) | `brand-book.html` | מסמך ייחוס שטוען `brand.css`. רכיב/טוקן חדש → קודם להוסיף ל‑`brand.css`, ולתעד כאן. |
| **ורטיקלים עיצוביים** (טפסי קשר, אקורדיון, טבלאות, טאבים, צעדים) | `design-verticals.html` (+ רכיבים ב‑`brand.css`) | דפוסים מורכבים מאושרים. דפוס חדש → להגדיר כאן, רכיביו ב‑`brand.css`. |
| **פונטים** | `fonts/fonts.css` + `--font-head` / `--font-body` | לכלול את ה‑CSS ולהשתמש במשתנים בלבד. |
| **לוגו ונכסי מותג** | `media/brand-logos/` (icon / lockup / wordmark · color/black/white) | להפנות לקובץ הקיים. אסור לשחזר לוגו בקוד/טקסט או להמציא וריאציה. |
| **תמונות אדם / הרצאות** | `media/adam/` (גרסאות web מותאמות: avif/webp + רספונסיב) | להשתמש בגרסאות ה‑`web/` המאופטמות. |
| **אייקונים** | Lucide (ראה למטה) | `data-lucide` בלבד. |
| **מבנה האתר** (עמודים, סקשנים, סדר, מטרות, טמפלטים) | `BLUEPRINT.md` (נערך ויזואלית ב‑`tools/blueprint.html`) | לבנות לפי המבנה. עמוד/סקשן חדש → לעדכן קודם את `BLUEPRINT.md`. |
| **קופי ומסרים** | `COPY.md` | לשאוב לפי קודים. אסור להמציא ניסוחים מקבילים. |
| **פרטי קשר** (טלפון, וואטסאפ, אימייל) | `contact.json` → אובייקט `SITE_CONFIG` | מקור אחד, לא לשכפל מספרים בקוד. |

חוקיות:
- **ברירת מחדל: לשאוב, לא להמציא.** לפני הוספת צבע/מידה/ניסוח/נכס — לבדוק אם כבר קיים במקור הרלוונטי ולהשתמש בו.
- **ערך חדש שהמשתמש מאשר** → להוסיף קודם למקור האמת (token / asset / COPY), ורק אז להשתמש בו דרך ההפניה.
- המטרה: קוד רזה, עקבי וקל לתחזוקה — שינוי גלובלי במקום אחד משנה את כל האתר.

## מבנה האתר (Blueprint) — עמודים וסקשנים

עץ האתר המלא — אילו **עמודים** קיימים, אילו **סקשנים** בכל עמוד, באיזה **סדר**, **מטרת** כל סקשן, ואילו סקשנים חולקים **טמפלט** — מרוכז ב‑**`BLUEPRINT.md`** (מקור אמת יחיד למבנה). העץ נערך ויזואלית בכלי הפנימי `tools/blueprint.html` (RTL: גרירה, שכפול, צביעת טמפלטים, ייצוא JSON/MD).

חוקיות:
- בבניית עמוד/סקשן — לעבוד לפי `BLUEPRINT.md` (מבנה, סדר, מטרות, טמפלטים). הקופי נשאב מ‑`COPY.md`, הקשר מ‑`contact.json`, העיצוב מ‑`brand.css`.
- עמוד/סקשן/טמפלט חדש שהמשתמש מאשר → **לעדכן קודם את `BLUEPRINT.md`**, ורק אז לבנות.
- כשהמשתמש עורך בכלי ומייצא `JSON` — לקלוט אותו ולעדכן בהתאם את `BLUEPRINT.md`.

## ספריית אייקונים

ספריית האייקונים של האתר היא **Lucide** (רישיון ISC, קוד פתוח וחינמי לשימוש מסחרי). ב‑`index.html` האייקונים **מוטמעים inline כ‑SVG** (pre-rendered מ‑Lucide v1.20) — ללא תלות CDN וללא JS בזמן ריצה, לטובת ביצועים/פרטיות/CLS. כל אייקון חדש חייב להיות מ‑Lucide, ולהיות מוטמע באותו פורמט (`<svg class="lucide icon" …>`). חריג: אייקוני מותג שאינם ב‑Lucide (וואטסאפ, וגם YouTube/Instagram/Facebook שהוסרו מ‑Lucide) — SVG ייעודי. עמודי הראווה עדיין יכולים להשתמש ב‑`<i data-lucide>` עם הסקריפט. ההשוואה שמורה ב‑`icon-options/`.

## תמונות ונכסים ויזואליים

חוקיות בשימוש בתמונות ובנכסים גרפיים באתר:

- **תמונות אדם (cutouts) קטומות בתחתית.** רוב התצלומים של אדם (`media/adam/web/cutouts/`) חתוכים בערך בגובה הירכיים. כדי שהקיטום לא ייראה מוזר/אקראי — **תמיד** להציב את התמונה כך שהקצה התחתון שלה יישב על "רצפת" הסקשן (צמודה לתחתית הסקשן / יושבת על קו ההפרדה לסקשן הבא), ו/או להוסיף **דעיכה רכה (gradient) לצבע הרקע בתחתית התמונה** או פס/בסיס עיצובי אחר שמטשטש את החיתוך. לעולם לא להשאיר תמונת אדם "מרחפת" עם קצה חתוך באמצע הסקשן. דוגמאות מיושמות: הירו הראשי (`.hero-portrait`, צמוד לתחתית) וסקשן "נדבר?" (`.contact-section .lead-figure`, צמוד לתחתית + דעיכת gradient).
- **נכסי assets והיפוך ל‑RTL.** חלק גדול מתמונות ה‑`media/assets/` נוצרו במקור לאתר LTR (אמריקאי) והאוריינטציה שלהן פונה שמאלה, מתוך הנחה שימוקמו בעמודה הימנית. באתר RTL שלנו הן יושבות לרוב **בעמודה השמאלית**, ולכן צריך **להפוך אותן אופקית** כדי שיפנו פנימה (לכיוון התוכן) — באמצעות מחלקת העזר **`.flip-x`** (`transform:scaleX(-1)`) שב‑`brand.css`. להפעיל שיקול דעת פר‑תמונה: להפוך רק נכסים שיש להם כיווניות ברורה ומוצבים בצד שמנוגד לה; לא להפוך נכסים סימטריים או כאלה שכוללים טקסט.

## טיפוגרפיה (פונטים)

הפונטים הרשמיים של הפרויקט — **בחירת המשתמש, ולשימוש בגרסה הסופית של האתר** (מקור הבחירה: סקשן ההירו ב‑`archive/variant-f-design-loop`):

- **כותרות / display: Rubik** (משקלים 500/600/700).
- **גוף הטקסט / body: IBM Plex Sans Hebrew** (משקלים 400/500/600/700).

הקבצים יושבים בתיקיית השורש של הפרויקט תחת `fonts/` ומשרתים את כל הגרסאות, כולל הסופית.

חוקיות:
- הפונטים **מתארחים מקומית** בתיקייה `fonts/` (לא מ‑CDN) — לטובת מהירות, פרטיות (ללא קריאות ל‑Google) ויציבות. הקבצים הם `woff2`, תת‑קבוצות **עברית + לטינית** בלבד, עם `font-display:optional` (מונע reflow/CLS בטעינה; בביקור חוזר מהמטמון מוצגים פונטי המותג מיד). ההגדרות ב‑`fonts/fonts.css`.
- כל שימוש בפונטים יהיה דרך `fonts/fonts.css` (לכלול אותו בעמוד), עם משתני CSS: `--font-head` ל‑Rubik (נופל ל‑IBM Plex Sans Hebrew) ו‑`--font-body` ל‑IBM Plex Sans Hebrew. אין להוסיף קישורי Google Fonts חדשים.
- שני הפונטים בקוד פתוח (Open Font License), חינמיים לשימוש מסחרי.

## קופי ומסרים (בנק מסרים)

ניסוחי הקופי המאושרים של האתר (סלוגנים, משפטי הירו, אותות אמון, ומסרים לכל שירות) מרוכזים ב‑**`COPY.md`** — מקור אמת יחיד, מאורגן בקודים `S/H/A/P/C/L`.

חוקיות:
- בכל בנייה/ניסוח של תוכן באתר — לשאוב מ‑`COPY.md` במקום להמציא ניסוחים מקבילים. מותר לקצר/להתאים פיסוק להקשר, אך לא לשנות את המסר.
- כל מסר כפוף לטרמינולוגיה: אסור "יועץ/ייעוץ השקעות" (פרט לשלילה בגילוי הנאות); להשתמש ב"משווק השקעות" / "בעל רישיון שיווק השקעות מטעם הרשות לניירות ערך" / "ליווי השקעות אישי".
- מסרים חדשים שהמשתמש מאשר — להוסיף ל‑`COPY.md` תחת הקטגוריה המתאימה.

## בדיקות ואיכות (לולאות סגורות)

התיקייה `qa/` מגדירה איך מוודאים שהאתר עובד מושלם תמיד: Unit + E2E (Playwright) + סשני AI חוקרים + Lighthouse, בלולאה סגורה מול KPI ברורים. נקודות כניסה: `qa/README.md` (מדיניות), `qa/user-stories.md` (מקור התרחישים, מתעדכן), `qa/kpi.md` (ספים), ו‑runbooks ל‑Playwright ול‑Lighthouse.

חוקיות ל‑Claude:
- **אחת לתקופה / לפני שינוי משמעותי — להציע למשתמש להריץ סשן בדיקות רוחבי.** כשהמשתמש מאשר או מבקש: קרא את `qa/README.md` + `user-stories.md` + `kpi.md`, הרץ לפי ה‑runbooks, מדוד מול הספים, תעד תוצאה תחת `qa/results/`, וכל ממצא שדורש תיקון → פתח כמשימה ב‑`TASKS.md` ותקן עד ירוק.
- כשה‑user stories מתעדכנים — לעדכן בהתאם את תרחישי ה‑E2E.

## ניהול משימות (Backlog)

הקובץ `TASKS.md` הוא ה-backlog של הפרויקט. חוקיות:

- כשמשימה בוצעה — לסמן אותה כבוצעה (`- [x] ...`) ולהעביר אותה מ"פעילות" אל "הושלמו" (בתחתית הקובץ).
